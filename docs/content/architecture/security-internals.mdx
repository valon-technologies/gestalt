# Security Internals

This page covers the implementation details of Gestalt's cryptographic operations, token lifecycle mechanics, and secret resolution flow. For operator-facing security guidance, see [Security](/security). For the datastore schema and encryption boundaries, see [Data Model](/architecture/data-model).

## Key derivation

`server.encryption_key` is converted to a 32-byte AES key at startup:

| Input | Derivation | Use case |
| --- | --- | --- |
| 64-character hex string | Hex-decoded directly to 32 bytes | Production. Generate with `openssl rand -hex 32`. |
| Any other string | Argon2id (3 iterations, 64 MiB, 4 threads, fixed salt `gestalt-derivekey-v1`) | Convenience for development. |
| Empty string | Error at runtime validation | Not allowed when `auth.provider` is not `none`. |

The derived key is used for all AES-256-GCM encryption and for signing session material. A 64-character hex key skips Argon2id entirely — if you already have 256 bits of entropy, key stretching adds startup cost without security benefit.

The Argon2id path uses a fixed salt, which means two deployments with the same passphrase produce the same key. Production deployments should always use a unique random hex key.

## AES-256-GCM encryption

Every encrypt operation:

1. Generates a fresh 12-byte nonce via `crypto/rand`
2. Encrypts with AES-256-GCM (authenticated encryption)
3. Prepends the nonce to the ciphertext
4. Base64-encodes the result for database storage

Decryption reverses the process: decode base64, split nonce from ciphertext, authenticate and decrypt. Tampered ciphertexts are detected and rejected.

All encrypted fields in the datastore use this same mechanism. There is no per-field or per-user key separation — a single derived key encrypts everything.

## Token lifecycle details

### Integration tokens (OAuth)

**Creation:**

1. User initiates OAuth via browser or CLI (`POST /auth/start-oauth`)
2. Gestalt generates an encrypted OAuth state parameter with a 10-minute TTL and optional PKCE verifier
3. User is redirected to the upstream provider's authorization URL
4. Provider redirects back with an authorization code
5. Gestalt exchanges the code for tokens, optionally with the PKCE verifier
6. Access token and refresh token are encrypted separately with AES-256-GCM
7. Tokens are upserted into `integration_tokens` keyed by `(user_id, integration, connection, instance)`

If the provider supports post-connect discovery (multiple resources like workspaces or accounts), Gestalt fetches the resource list after the token exchange. If more than one resource is found, the user selects one, and the selection is stored in `metadata_json`.

**Manual connections** follow the same storage path — the user-provided credential becomes the `access_token_encrypted` value with an empty refresh token.

**Refresh:**

Token refresh happens on-demand during operation invocation, not as a background job:

1. Before dispatching an operation, the broker checks `expires_at`
2. If the token expires within 5 minutes and a refresh token exists, refresh is attempted
3. Concurrent refresh attempts for the same token are deduplicated (singleflight)
4. On success: new access token (and optionally new refresh token) encrypted and stored, `refresh_error_count` reset to 0
5. On failure: `refresh_error_count` incremented. If the token is still technically valid, it is used with a warning. If expired, the invocation fails with an error

The `refresh_error_count` persists across requests so operators can detect tokens stuck in a refresh-failure loop.

**Revocation:**

Integration tokens are deleted (not soft-deleted) via:

- CLI: `gestalt integrations disconnect NAME [--connection ...] [--instance ...]`
- HTTP API: `DELETE /api/v1/integrations/{name}[?connection=...&instance=...]`

The encrypted token data is removed from the database. There is no upstream token revocation — Gestalt does not call the provider's revoke endpoint.

### API tokens

**Creation:**

1. 32 random bytes generated via `crypto/rand`
2. Plaintext formatted as `gst_api_` + hex encoding (total ~73 characters)
3. SHA-256 hash computed and stored in `api_tokens.hashed_token`
4. Plaintext returned to the caller exactly once

**Validation:**

On each request with a `Bearer gst_api_*` header:

1. Compute SHA-256 hash of the incoming token
2. Look up the hash in `api_tokens` where `expires_at IS NULL OR expires_at > now()`
3. If found, resolve the associated user and grant access with the token's scopes

**Revocation:**

- Single token: `DELETE /api/v1/tokens/{id}` — deletes the row
- All tokens: `DELETE /api/v1/tokens` — deletes all rows for the authenticated user

Revocation is immediate. There is no grace period or propagation delay.

### Platform sessions

Sessions are cookie-based and managed by the auth provider, not stored in the Gestalt datastore:

1. User completes the identity provider login flow
2. Gestalt validates the callback (checks `email_verified` for Google and OIDC)
3. A session cookie is set with the auth provider's token
4. The cookie is validated on each subsequent request by the auth provider
5. Logout clears the cookie (`MaxAge=-1`); there is no server-side session store to invalidate

## Secret resolution flow

```
config.yaml on disk
  │
  ├─ ${ENV_VAR} expansion (+ _FILE fallback)
  │
  ├─ YAML decode
  │
  ├─ Apply defaults (port 8080, secrets: env, telemetry: stdout)
  │
  ├─ Structural validation
  │
  ├─ Build secret manager from secrets.config
  │    (secrets.config uses ${ENV_VAR} only, not secret://)
  │
  ├─ Resolve secret:// references across all other config sections:
  │    server, auth.config, datastore.config, telemetry.config,
  │    audit.config, and all provider configs
  │
  ├─ Runtime validation (encryption_key, auth.provider, datastore.provider required)
  │
  └─ Proceed to component initialization
```

Secrets are resolved once at startup. If a secret value changes in the external secret manager, Gestalt must be restarted to pick up the new value. There is no runtime secret refresh.

## Key rotation

There is no automated key rotation. To rotate `server.encryption_key`:

1. Stop all Gestalt instances
2. Decrypt all tokens with the old key and re-encrypt with the new key (requires direct database access and the encryption/decryption logic from `core/crypto`)
3. Update the config with the new key
4. Restart

This is a known limitation. In practice, key rotation is a rare event — the more common operational need is ensuring the key is properly backed up and not accidentally changed.

# Authorization Model

Gestalt separates four concerns that are easy to conflate:

| Concern | What it answers | Where it is configured |
| --- | --- | --- |
| Authentication | Who is calling? | `providers.authentication`, sessions, API tokens, workflow execution refs |
| Operation scope | Which plugin operations may this principal call? | API token `permissions`, workflow execution refs, config-managed workflow `permissions` |
| Runtime authorization | Is this subject allowed under the deployment policy? | `providers.authorization`, plugin/UI `authorizationPolicy` |
| Credentials | Which upstream token or platform credential should be used? | Plugin `connections`, connection mode, connected user credentials |

These layers are checked independently. Granting one does not imply the others.
For example, `plugins.brain.invokes` can allow Brain to attempt a Slack call,
but the execution principal still needs `slack.<operation>` scope, and the
Slack operation still resolves its own connection and credential.

## Request Principals

Every invocation reaches the broker with a principal. The principal usually
comes from a session cookie or API token. Delayed work uses an execution
reference instead: the workflow provider stores the owner subject, auth source,
credential subject, target, and permission ceiling when the schedule or trigger
is created.

API tokens and execution refs can carry operation-scoped permissions. When
`TokenPermissions` are present, Gestalt treats them as a ceiling. The caller
must have the target plugin and, when operations are listed, the exact target
operation.

## Broker Checks

The broker is the common runtime gate for HTTP, MCP, workflow, agent-tool, and
nested plugin invocations. For a plugin operation call it:

1. Resolves the target plugin and operation.
2. Requires the principal to include the target plugin.
3. Requires the principal to include the target operation when operation scopes
   are present.
4. Resolves the canonical subject and applies the configured authorization
   provider/policy when one is bound.
5. Resolves the connection and credential for the target operation.
6. Invokes the provider with only the token needed for that invocation.

The authorization provider does not replace operation scopes. It is the
deployment policy layer. Operation scopes still decide whether the principal is
allowed to call `slack.conversations.history` at all.

## Nested Plugin Invocations

Executable plugins can call other plugins through the host-provided plugin
invoker. This is intentionally a propagation and attenuation primitive, not an
authority-expansion primitive.

When Gestalt starts a plugin that declares `invokes`, it gives that plugin a
host service and a short-lived invocation token. The token carries the current
principal, request metadata, invocation depth, credential context, and the
declared invocation grants for that caller plugin.

On a nested call, Gestalt checks both:

| Check | Purpose |
| --- | --- |
| `plugins.<caller>.invokes` | The caller plugin is allowed to attempt this dependency. |
| Propagated principal permissions | This execution is allowed to call the target operation. |

Invocation tokens can be exchanged only for subsets of their parent grants.
That lets plugins narrow delegated authority, but not add new target plugins or
operations. After token validation, Gestalt restores the propagated principal
and sends the nested call back through the normal broker checks.

This split is deliberate. If `invokes` automatically granted runtime operation
scope, every run of a plugin would inherit every downstream operation that the
plugin package declares. Instead, `invokes` defines the allowed call graph, and
the principal or workflow execution ref defines what this specific execution
may do.

## Workflows

User-created schedules and triggers execute under the subject that created
them. At creation time, Gestalt stores an execution ref with the target and the
creator's permission ceiling. When the workflow fires later, Gestalt recreates
the principal from that execution ref and invokes the target through the
broker.

Config-managed workflows use the same execution-ref model, but the subject is
`system:config`. The direct target operation is included automatically. If the
target performs nested plugin calls, declare those additional operation scopes
with `permissions`:

```yaml
workflows:
  schedules:
    brain_slack_sync:
      provider: local
      cron: "*/15 * * * *"
      target:
        plugin:
          name: brain
          operation: sources.sync
          input:
            sourceId: slack-valon-public
      permissions:
        - plugin: slack
          operations:
            - conversations.list
            - conversations.history
            - conversations.replies
```

This grants the workflow execution principal `brain.sources.sync` plus the
listed Slack operation scopes. It does not grant Slack credentials. Slack still
uses the connection selected by the Slack operation and deployment config.

Config-owned direct targets, agent tools, and output-delivery targets must
resolve to non-user credentials. Mixed plugins are still supported when the
target explicitly selects a platform-backed connection.

## Debugging Denials

The layer that failed is usually visible from the error:

| Symptom | Likely missing piece |
| --- | --- |
| `plugin "brain" may not invoke slack.conversations.history` | `plugins.brain.invokes` is missing or too narrow. |
| `scope denied: slack.conversations.history` | The propagated principal or workflow execution ref lacks the Slack operation scope. |
| `authorization denied` | The authorization provider or bound policy rejected the subject. |
| `not authenticated` or missing upstream token | The target operation resolved to a connection with no usable credential for the credential subject. |

For config-managed source syncs, the common fix is to add explicit workflow
`permissions` for the downstream plugin operations that the source sync will
call, while keeping upstream credentials on platform-backed connections.

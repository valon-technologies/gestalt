# Datastores

Datastore providers back the persistent state layer. Gestalt stores users, integration tokens, API tokens, and OAuth registrations through the datastore interface, keeping the storage backend fully replaceable. The first-party SQLite provider works for single-node deployments. For production or multi-replica setups, use Postgres, MySQL, or another datastore provider.

The host encrypts all integration tokens before passing them to the datastore using AES-256-GCM, so the datastore stores sealed bytes without needing access to the encryption key. API tokens are stored as one-way SHA-256 hashes. User emails and display names are stored in plaintext.

## Manifest

A datastore provider manifest declares `kinds: [datastore]`:

```yaml
source: github.com/example/plugins/my-datastore
version: 0.0.1-alpha.1
display_name: My Datastore
description: Custom persistent storage backend
kinds:
  - datastore
datastore:
  config_schema_path: ./datastore_config.json
```

The optional `datastore.config_schema_path` field points to a JSON Schema that validates the `datastore.config` block in the server config. If you provide one, Gestalt rejects invalid config at startup.

## Interface

A datastore provider is a larger interface than auth or integration providers. It composes user storage, integration token storage, and API token storage, plus `Migrate` for schema setup and `HealthCheck` for readiness probes.

    	ds.db, err = sql.Open("postgres", dsn)

    }

    	return ds.db.PingContext(ctx)

    	// Create users, integration_tokens, and api_tokens tables.
    	return nil

    	gestalt.ServeDatastoreProvider(context.Background(), &MyDatastore{})

## OAuthRegistrationStore

If your deployment uses MCP OAuth with dynamic client registration, the datastore needs to store OAuth client registrations. This is an optional interface. Implement `GetOAuthRegistration`, `PutOAuthRegistration`, and `DeleteOAuthRegistration` in Go, inherit from `gestalt.OAuthRegistrationStore` in Python, or override the default methods in Rust.

Deployments that do not use MCP OAuth can skip this entirely. The default implementations return unimplemented errors, which is the correct behavior when MCP OAuth is not configured.

## Config reference

To use your datastore provider in a Gestalt deployment, reference it in the `datastore` block:

```yaml
datastore:
  provider:
    source:
      path: ./my-datastore/plugin.yaml
  config:
    dsn: ${DATABASE_URL}
```

`provider.source.path` points at the datastore provider's `plugin.yaml` during development. For production, use `provider.source.ref` and `version` to reference a published release. The `config` block is passed to `Configure` at startup and validated against the config schema if one was declared in the manifest.

Gestalt calls `Migrate` once at startup before any other datastore methods. Use it to create tables, run migrations, or verify that the schema is up to date. `HealthCheck` is called periodically and should return quickly; a database ping is the typical implementation.

# Web UIs

Web UI bundles are static asset packages. Gestalt serves them under explicit path prefixes configured in `providers.ui`. The built-in admin UI at `/admin` remains available regardless of whether custom UI bundles are configured.

A UI bundle is not a plugin in the executable sense. It contains no server-side code. Gestalt simply serves the static files from the asset root directory. The bundle talks to Gestalt through the same HTTP API and MCP endpoints that any other client would use.

## Manifest

The manifest declares `spec.assetRoot`, which points to the directory containing the built static assets. The source field identifies the provider in the registry.

```yaml
kind: webui
source: github.com/valon-technologies/gestalt-providers/web/default
version: 0.0.1
displayName: Default Web UI
description: Static Gestalt web UI bundle.
release:
  build:
    command:
      - sh
      - ./build.sh
spec:
  assetRoot: out
```

The `assetRoot` path is relative to the manifest file. It should contain the output of your frontend build (typically an `index.html` and accompanying JS, CSS, and image files).

## Build step

If your UI needs a build step before packaging, use `release.build` to run it automatically during `gestaltd provider release`. This runs before source-manifest preparation, so the built assets are included in the release archive.

This example uses a build step that runs `npm run build` to produce the static output:

```yaml
kind: webui
source: github.com/valon-technologies/gestalt-providers/web/default
version: 0.0.1
release:
  build:
    command:
      - npm
      - run
      - build
spec:
  assetRoot: out
```

The command runs relative to the manifest directory. After the build completes, Gestalt packages everything under `assetRoot` into the release archive. You can use any build tool here -- `npm run build`, `vite build`, `next build && next export` -- as long as it writes output to the directory specified in `assetRoot`.

## Configuring mounted UI bundles

Reference a UI bundle under `providers.ui.<name>`. During development, point at the source manifest:

```yaml
providers:
  ui:
    roadmap:
      source:
        path: ./customer-roadmap-review/ui/manifest.yaml
      path: /create-customer-roadmap-review
```

For production, reference a published release by source and version:

```yaml
providers:
  ui:
    roadmap:
      source:
        ref: github.com/your-org/web/customer-roadmap-review
        version: 0.0.1
      path: /create-customer-roadmap-review
```

When a config references a published UI bundle, run `gestaltd init` to resolve and prepare it. This writes lock state to `gestalt.lock.json` and extracts the assets under `.gestaltd/`. After init, `gestaltd serve --locked` serves the locked UI bundle under its configured path prefix. If the prepared UI files are already present, Gestalt uses them directly; if they are missing, Gestalt can materialize them from the lockfile at startup.

Gestalt serves each bundle for requests under `<path>/*`. Mounted bundles are static SPAs; they talk to Gestalt through the same HTTP API and MCP endpoints, not through a special plugin-local route surface.

## The admin UI

The built-in admin UI is always served at `/admin`, regardless of whether custom UI bundles are configured. It provides a management interface for users, plugins, connections, and API tokens. UI bundles own only their configured prefixes.

## Release

UI bundles use the same release system as provider packages:

```sh
gestaltd provider release --version 0.0.1
```

Run this from the directory containing the manifest. If `release.build` is defined, the build command runs first, then Gestalt packages the manifest and the asset root into a release archive. See [Releasing](/providers/releasing) for details on release archives, tag resolution, and prepared state.

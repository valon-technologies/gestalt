# Configuration

Gestalt is configured from a single YAML file. The [Config File](/reference/config-file) reference covers every field and its validation rules; this page explains the model and the main workflows.

## Config lifecycle

When `--config` is not supplied, `gestaltd` looks for config in this order:

1. `GESTALT_CONFIG`
2. `./config.yaml`
3. `~/.gestaltd/config.yaml`
4. `~/.gestalt/config.yaml`
5. `/etc/gestalt/config.yaml`

If nothing is found, `gestaltd` generates a default local config at `~/.gestaltd/config.yaml`.

Gestalt expands `${ENV_VAR}` placeholders before YAML decode, falling back to `ENV_VAR_FILE` when the direct variable is unset. Unknown fields are rejected. Relative paths resolve from the config file directory, and `secret://` references are resolved through the configured secret manager at bootstrap.

## Top-level blocks

A config file is organized into a handful of top-level blocks: `server` (HTTP listener, base URL, encryption key, API token TTL, prepared artifacts dir), `auth` (platform login for the web UI, HTTP API, and `/mcp`), `datastore` (persistent storage for users, sessions, API tokens, and integration credentials), `secrets` (resolution for `secret://` references), `telemetry` (structured logs, metrics, traces, and OTLP export), `providers` (the integrations users actually invoke), and optionally `egress` (outbound policy rules and credential grants) and `ui` (packaged web UI bundle). The full example and minimal config at the bottom of this page show the structure in practice.

## Defaults and required fields

Gestalt defaults `server.port` to `8080`, `secrets.provider` to `env`, and `telemetry.provider` to `stdout`. You must always set `auth.provider`, `datastore.provider`, and `server.encryption_key`.

`server.encryption_key` is the root deployment secret. Gestalt derives token encryption and related session material from it, so changing it is a meaningful deployment event rather than a cosmetic config update.

## Providers

Each entry under `providers:` becomes a runtime provider. A provider can be assembled from inline REST operations in YAML, an OpenAPI document, a GraphQL endpoint, an upstream MCP server, or a local or published [plugin source](/plugins). A single provider can mix executable plugin logic with spec-backed or MCP surfaces, which is the main composition model in Gestalt.

## Provider patterns

An inline REST provider defines operations directly in YAML.

```yaml
providers:
  support:
    connections:
      default:
        auth:
          type: bearer
          credentials:
            - name: token
              label: API Token
    surfaces:
      rest:
        base_url: https://api.support.example.com
        operations:
          - name: list_tickets
            description: Return open support tickets
            method: GET
            path: /tickets
```

An OpenAPI-backed provider references an external spec document and can selectively expose operations.

```yaml
providers:
  httpbin:
    connections:
      public:
        mode: none
        auth:
          type: none
    surfaces:
      openapi:
        document: ./openapi/httpbin.yaml
        connection: public
    allowed_operations:
      get_ip:
        alias: get_ip
    mcp:
      enabled: true
      tool_prefix: httpbin_
```

A local source plugin loads its definition from a plugin manifest on disk.

```yaml
providers:
  gmail:
    display_name: Gmail
    from:
      source:
        path: ./plugins/gmail/plugin.yaml
    mcp:
      enabled: true
      tool_prefix: gmail_
```

A hybrid provider combines a plugin source with additional surfaces like OpenAPI specs and upstream MCP servers.

```yaml
providers:
  support:
    from:
      source:
        path: ./plugins/support/plugin.yaml
    connections:
      mcp:
        mode: user
        auth:
          type: mcp_oauth
    surfaces:
      openapi:
        document: ./openapi/support.yaml
      mcp:
        url: https://mcp.support.example.com/mcp
        connection: mcp
    allowed_operations:
      tickets.list:
        alias: list_tickets
```

## Connections and auth

Connections define how Gestalt authenticates to upstream systems.

### Connection modes

**none** means no upstream credential is needed. **user** means each user stores their own. **identity** uses one shared service credential. **either** prefers user, falls back to identity.

### Connection naming

`connections.default` is common but not required. If you use only named connections, each configured surface must explicitly name which connection it uses. Users can also store multiple instances of the same connection; the [HTTP API](/reference/http-api) exposes `_connection` and `_instance` selectors so callers can disambiguate which stored credential to use.

### Auth types

Common connection auth types are `oauth2`, `mcp_oauth`, `bearer`, `manual`, and `none`.

Gestalt stores upstream credentials by user, provider, connection name, and instance. Renaming a `providers:` key is therefore a breaking change for stored connections.

### Choosing an auth strategy

For local development, set `auth.provider: none`. A single-user dev server that needs login flows can use `auth.provider: local`, while real multi-user deployments should use `auth.provider: google` or `auth.provider: oidc`. On provider connections, set `mode: user` when each person should authenticate with their own upstream credential, or `mode: identity` when a single shared service credential is appropriate.

## Platform auth and API tokens

Platform auth controls access to the web UI, HTTP API, and `/mcp`. Set `auth.provider: none` for local development, `auth.provider: local` for a single-user dev environment, or `auth.provider: google` / `auth.provider: oidc` for multi-user setups. See [Security](/security) for trust boundaries and token handling details.

For programmatic access, Gestalt issues API tokens prefixed with `gst_api_`. They are accepted through `Authorization: Bearer ...` and use the same auth middleware as browser sessions.

## MCP

Gestalt exposes a single MCP endpoint at `/mcp`, using the same auth model as the rest of the HTTP surface. Control tool naming with the provider-level `mcp` block.

```yaml
providers:
  github:
    mcp:
      enabled: true
      tool_prefix: github_
```

Tool exposure follows the provider surfaces and catalog that Gestalt builds at startup. Declarative REST, OpenAPI, GraphQL, upstream MCP, and plugin-backed operations can all contribute tools.

## Managed parameters and response mapping

Use `managed_parameters` when OpenAPI- or GraphQL-backed operations need fixed header or path values injected before they are exposed to callers.

```yaml
managed_parameters:
  - in: header
    name: X-API-Version
    value: "2026-04-01"
  - in: path
    name: account_id
    value: primary
```

Use `response_mapping` when the upstream response needs explicit data or pagination extraction.

```yaml
response_mapping:
  data_path: data.items
  pagination:
    has_more_path: pageInfo.hasNextPage
    cursor_path: pageInfo.endCursor
```

## Startup workflows

For fast local development, run the server directly.

```sh
gestaltd serve --config ./config.yaml
```

For deterministic production startup, prepare locked artifacts first.

```sh
gestaltd init --config ./config.yaml
gestaltd serve --locked --config ./config.yaml
```

To validate a config without starting the server, use the validate command.

```sh
gestaltd validate --config ./config.yaml
```

`init` prepares published [plugin](/plugins) sources, packaged UI bundles, and other locked artifacts into writable state. `serve --locked` starts only from that prepared state.

## Prepared state and lockfiles

Gestalt supports two startup modes: mutable startup with `gestaltd serve`, and deterministic startup with `gestaltd init` followed by `gestaltd serve --locked`.

`gestaltd init` writes `gestalt.lock.json` next to the config file, along with prepared provider and UI artifacts under `.gestaltd/` in the configured artifacts directory. A typical layout looks like this:

```text
deploy/
  config.yaml
  gestalt.lock.json
  .gestaltd/providers/...
  .gestaltd/ui/...
```

The config declares intent, while the lockfile records the exact prepared result that `serve --locked` should trust. That lets Gestalt verify that prepared artifacts still match the current config instead of just using whatever happens to exist on disk.

### Lockfile example

For a source-backed plugin, the YAML config says which release you intend to use.

```yaml
providers:
  example:
    from:
      source:
        ref: github.com/acme/plugins/example
        version: 1.2.3
```

The lockfile records the concrete prepared result of resolving that config.

```json
{
  "version": 1,
  "providers": {
    "example": {
      "fingerprint": "...",
      "source": "github.com/acme/plugins/example",
      "version": "1.2.3",
      "resolved_url": "https://github.com/...",
      "archive_sha256": "...",
      "manifest": ".gestaltd/providers/example/plugin.yaml",
      "executable": ".gestaltd/providers/example/artifacts/linux/amd64/provider"
    }
  },
  "ui": {
    "fingerprint": "...",
    "source": "github.com/acme/plugins/web",
    "version": "1.2.3",
    "resolved_url": "https://github.com/...",
    "archive_sha256": "...",
    "manifest": ".gestaltd/ui/plugin.yaml",
    "asset_root": ".gestaltd/ui/dist"
  }
}
```

That distinction matters because `serve --locked` needs a durable record of what was prepared during `init`, not just whatever files happen to exist on disk.

## Example

This config shows a production-style deployment with OIDC auth, a GitHub provider backed by both OpenAPI and upstream MCP, and per-user connections.

```yaml
server:
  base_url: https://gestalt.example.com
  encryption_key: secret://gestalt-encryption-key

auth:
  provider: oidc
  config:
    issuer_url: https://login.example.com
    client_id: ${OIDC_CLIENT_ID}
    client_secret: secret://oidc-client-secret

providers:
  github:
    display_name: GitHub
    mcp:
      enabled: true
      tool_prefix: github_
    connections:
      api:
        mode: user
        auth:
          type: bearer
          credentials:
            - name: token
              label: Personal Access Token
      mcp:
        mode: user
        auth:
          type: mcp_oauth
    surfaces:
      openapi:
        document: ./openapi/github.yaml
        connection: api
      mcp:
        url: https://api.githubcopilot.com/mcp/
        connection: mcp
```

## Minimal config

The smallest useful config sets up a local server with SQLite storage and no auth. See [Observability](/observability) for adding metrics, logs, and OTLP export.

```yaml
server:
  port: 8080
  base_url: http://localhost:8080
  encryption_key: change-me

auth:
  provider: none

datastore:
  provider: sqlite
  config:
    path: ./gestalt.db

providers: {}
```

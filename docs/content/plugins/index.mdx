# Plugins

Gestalt uses one packaging system for executable provider plugins, declarative provider packages, and packaged web UI bundles. This section covers the cases where you need to go beyond inline YAML.

## When you need a plugin

Use inline config when a provider can be described as declarative REST operations, an OpenAPI document, a GraphQL endpoint, or an upstream MCP server. Use an executable plugin when the provider needs logic that those formats cannot express: custom request or response shaping, multi-step workflows, proprietary protocols, or request-scoped catalogs driven by runtime state.

## How plugins work

When a provider entry includes a `from` block that resolves to an executable
plugin, `gestaltd` starts the plugin as a child process. The host creates a
temporary Unix socket, passes the socket path through
`GESTALT_PLUGIN_SOCKET`, and the plugin connects back over gRPC to register
itself.

The host still owns auth, token storage, and request routing. The plugin
receives only its configured provider name, provider config, and the current
request context it needs to execute operations.

Plugins run in process isolation. They cannot access the datastore, other users' credentials, or the host process memory. The host resolves credentials and passes only the current caller's access token to the plugin for each request.

## SDK support

Gestalt ships first-party SDKs for Go (`sdk/go`) and Python (`sdk/python`). The underlying protocol is gRPC, so other languages are possible, but the documented authoring flow is centered on those two.

## Packaging model

The same manifest and release format packages declarative providers (no binary), executable plugins with bundled artifacts, and web UI bundles referenced through `ui.plugin`. See [Plugin Manifests](/reference/plugin-manifests) for the manifest schema, [Writing a Plugin](/plugins/writing-a-plugin) for the end-to-end authoring flow, and [Packaging and Releasing](/plugins/packaging-and-releasing) for release archives.

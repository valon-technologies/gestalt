# Plugins

Gestalt uses one packaging system for three related extension types:

- executable provider plugins
- declarative provider packages
- packaged web UI bundles

This section focuses on the cases where you need to go beyond inline YAML.

## When you need a plugin

Use inline config when a provider can be described as:

- declarative REST operations
- an OpenAPI document
- a GraphQL endpoint
- an upstream MCP server

Use an executable plugin when the provider needs logic that cannot be expressed
that way, such as:

- custom request or response shaping
- multi-step workflows
- proprietary protocols
- request-scoped catalogs driven by runtime state

## How plugins work

When a provider entry includes a `from` block that resolves to an executable
plugin, `gestaltd` starts the plugin as a child process. The host creates a
temporary Unix socket, passes the socket path through
`GESTALT_PLUGIN_SOCKET`, and the plugin connects back over gRPC to register
itself.

The host still owns auth, token storage, and request routing. The plugin
receives only its configured provider name, provider config, and the current
request context it needs to execute operations.

Plugins run in process isolation. They cannot access the datastore, other users' credentials, or the host process memory. The host resolves credentials and passes only the current caller's access token to the plugin for each request.

## SDK support

Gestalt currently ships first-party SDKs for:

- Go (`sdk/go`)
- Python (`sdk/python`)

The underlying protocol is gRPC, so other languages are possible, but the
documented authoring flow today is centered on the Go and Python SDKs.

## Packaging model

The same manifest and release format can package:

- a declarative provider with no executable binary
- an executable provider plugin with bundled artifacts
- a web UI bundle referenced through `ui.plugin`

See [Plugin Manifests](/reference/plugin-manifests) for the exact manifest
syntax.

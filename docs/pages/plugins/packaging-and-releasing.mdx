# Packaging and Releasing

Once a plugin is working locally with `from.source.path`, you can publish release archives for distribution. Release archives are self-contained and include a manifest, compiled binaries, and optional assets.

## Source manifest vs release manifest

Every release archive requires a manifest file named `plugin.yaml` (or `plugin.yml` or `plugin.json`). During local development and `gestaltd plugin release`, pure executable source manifests can stay lean and omit `artifacts` and `entrypoints.provider`. The release command materializes those fields for you.

A minimal source manifest for an executable provider:

```yaml
source: github.com/acme/plugins/my-plugin
version: 0.1.0
kinds:
  - provider
provider:
  connections:
    default:
      auth:
        type: none
```

For Go source plugins, the executable code lives under `provider/`. You do not write `main.go`; Gestalt synthesizes the wrapper during source execution and release packaging.

### Declarative-only release archives

A manifest can describe a declarative provider that requires no compiled binary. This is useful for distributing reusable REST providers, OpenAPI-backed integrations, or web UI bundles:

```yaml
name: my-rest-provider
version: 0.1.0
kinds:
  - provider
```

## Build a cross-platform release

Use `gestaltd plugin release` to compile the plugin for multiple platforms and produce release archives:

```sh
gestaltd plugin release --version 0.1.0
```

Run this from the plugin source directory. It compiles the binary for each target platform, produces release archives, and writes a checksums file.

For executable Go plugins, release materializes the executable catalog
automatically from the provider's typed router before packaging.

## Reference a plugin

There are two ways to reference a plugin from the Gestalt config:

### Local source path

Use the source tree directly during development:

```yaml
providers:
  my-plugin:
    from:
      source:
        path: ./plugins/my-plugin/plugin.yaml
```

### Versioned source

Reference a published plugin by source and version. `gestaltd init` resolves the release archive and writes it to lock state:

```yaml
providers:
  my-plugin:
    from:
      source:
        ref: github.com/your-org/your-plugins/my-plugin
        version: 0.1.0
```

This is the recommended approach for production deployments.

## Prepared state

When a config references published plugins, run `gestaltd init` to resolve and prepare them:

```sh
gestaltd init --config ./config.yaml
```

This writes lock state (`gestalt.lock.json`) and extracts prepared artifacts under `.gestaltd/`. After init, `gestaltd serve --locked` starts the server using only the prepared state, without fetching or resolving anything at runtime. See [Configuration](/concepts/configuration) for the full startup model.

## Release archive layout

A typical release archive contains:

```
plugin.yaml
artifacts/
  my-plugin-linux-amd64
  my-plugin-darwin-arm64
schemas/
  config.schema.json
assets/
  icon.svg
```

The exact structure is determined by the manifest. The `schemas/` directory is optional and used for config validation. The `assets/` directory is optional and used for icons or other static files.

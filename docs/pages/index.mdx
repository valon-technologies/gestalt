# Gestalt

`gestaltd` is a self-hosted integration gateway. One YAML file defines how the
server authenticates users, stores credentials, resolves secrets, loads
integrations, and exposes them through a single HTTP API and optional `/mcp`
endpoint.

## What It Covers

- One runtime for REST, GraphQL, MCP, packaged providers, runtimes, and HTTP
  bindings.
- One connection model for OAuth, manual credentials, bearer tokens, shared
  identities, and no-auth integrations.
- One packaging model for provider plugins, runtime plugins, and web UI
  plugins.
- One deployment workflow for local development and locked production startup.

## The Main Building Blocks

| Piece | Purpose |
| --- | --- |
| `gestaltd` | The server binary. It loads config, starts the API and UI, and mounts `/mcp` when integrations expose tools. |
| `config.yaml` | The control plane for auth, datastore, secrets, telemetry, integrations, runtimes, bindings, egress, and UI plugins. |
| `gestalt.lock.json` and `.gestalt/` | Prepared state written by `gestaltd init` for packaged integrations, runtimes, and UI plugins. |
| `plugin.yaml` | A plugin manifest for packaged providers, runtimes, or UI bundles. |
| `gestalt` | The client CLI for login, discovery, invocation, and API token management. |

## Common Workflows

- Run `gestaltd` with no config for a local server with generated SQLite state.
- Write an explicit `config.yaml` with inline integrations or packaged plugins.
- Use `gestaltd init` to prepare plugin artifacts, then `gestaltd serve --locked`
  for deterministic startup.
- Connect integrations once and use the same provider graph from the HTTP API,
  the embedded UI, or `/mcp`.

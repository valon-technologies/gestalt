# Contributing

The repository is laid out to make extension work obvious: core interfaces, internal runtime packages, plugin implementations, clients, and deployment surfaces are separated cleanly.

## Repository layout

- **`cmd/toolshed`** — the main Go binary entry point. Registers all plugins and starts the server.
- **`core`** — public interfaces for auth providers, datastores, secret managers, and integration hooks.
- **`internal`** — runtime packages: config loading, HTTP routing, middleware, operation dispatch, and token management.
- **`plugins`** — implementations of core interfaces: auth providers, datastores, secret managers, and integration-specific hooks.
- **`client/cli`** — the Rust CLI source (`tshed`).
- **`web`** — the Next.js web UI.

## How to add a new plugin type implementation

1. Implement the interface defined in `core` (e.g. `core.AuthProvider`, `core.Datastore`, `core.SecretManager`).
2. Expose a factory function that accepts the provider config map and returns an initialized instance.
3. Register the factory in `cmd/toolshed/main.go` so the runtime can resolve it by name from config.
4. Add tests that exercise the plugin through the same interface the runtime uses.

## How to add integration-specific behavior

Integration hooks live under `plugins/integrations`. Each package exports named hook functions that are registered in `main.go`. To add behavior for a new upstream API, create a package under `plugins/integrations`, implement the relevant hook signatures (`response_check`, `token_parser`, or `request_mutator`), and register them by name.

## Development workflow

```sh
cd toolshed
go test ./...

cd web
npm run check
npm run build
```

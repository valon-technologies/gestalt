# Plugin Protocol

Packaged provider and runtime plugins speak a small gRPC protocol. Most plugin
authors should use the Go SDK instead of implementing the wire protocol by hand.

## Provider Plugins

A provider plugin is responsible for:

- reporting metadata
- exposing a catalog of operations
- executing an operation when the host asks for it

In the Go SDK, that means implementing:

- `Name`
- `DisplayName`
- `Description`
- `ConnectionMode`
- `Catalog`
- `Execute`

Optional provider interfaces add:

- startup hooks
- dynamic session-scoped catalogs
- manual auth support
- connection parameter definitions

## Runtime Plugins

A runtime plugin is long-lived. It starts once, receives the set of available
capabilities, and can call back into Gestalt through the runtime host.

In the Go SDK, a runtime implements:

- `Start`
- `Stop`

The runtime host provides:

- `Invoke`
- `ListCapabilities`

## Catalogs Are The Contract

Gestalt does not ask provider plugins to register HTTP routes. Instead, provider
plugins publish a catalog and Gestalt projects that catalog into:

- the HTTP API
- the embedded UI
- `/mcp`

That keeps packaged providers consistent with inline declarative integrations.

## Sockets And Lifecycle

The host starts packaged plugins over a Unix socket whose path is provided
through environment variables. The Go SDK helpers:

- `ServeProvider`
- `ServeRuntime`

handle socket setup, gRPC registration, and graceful shutdown.

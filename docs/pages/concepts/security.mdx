# Security

Gestalt's security model is mostly about separation of concerns:

- platform auth secures the server itself
- integration auth secures outbound calls to upstream systems
- egress policy constrains where bindings and upstream clients can talk
- secrets providers keep runtime secrets out of static config where possible

## Server-Side Controls

### Platform Auth

Use `google` or `oidc` for normal multi-user deployments. `none` is appropriate
for local development and tightly controlled environments.

### Encryption Key

`server.encryption_key` is required whenever platform auth is enabled. It is
used to derive keys for state encryption and secure token handling.

### API Tokens

API tokens are bearer credentials to Gestalt itself. Use scopes to limit the
integrations they can call, and set `server.api_token_ttl` when you want
automatic expiration.

## Integration Credentials

Integration credentials are stored separately from platform auth:

- OAuth tokens are stored after a successful connect flow.
- Manual credentials are stored through `connect-manual`.
- Connection params are validated before the connection is created.

## Egress Policy

Bindings and upstream HTTP clients can be restricted under `egress`.

```yaml
egress:
  default_action: deny
  policies:
    - action: allow
      subject_kind: system
      subject_id: support-proxy
      host: api.example.com
      path_prefix: /v1
```

You can also inject outbound credentials with `egress.credentials` and
`secret://...` references.

## Secrets Providers

Built-in secret managers include:

- `env`
- `file`
- `google_secret_manager`
- `aws_secrets_manager`
- `vault`
- `azure_key_vault`

Use them when runtime features need secret references instead of plain config
values.

## Cookies And TLS

Gestalt marks session cookies `Secure` when `server.base_url` uses `https://`.
In practice, production deployments should terminate TLS in front of Gestalt and
set `server.base_url` to the public HTTPS URL.

## Proxy And Webhook Bindings

- Proxy bindings require auth unless the binding itself is explicitly public.
- Proxy routes use egress policy and can add outbound credentials through egress
  grants.
- Webhook bindings support `public`, `signed`, and `trusted_user_header`
  authentication modes.

# Platform Model

Gestalt has a small set of primitives. Once those are clear, the rest of the
system is mostly composition.

## One Config, Many Surfaces

The server starts from a single YAML config and turns it into:

- a platform auth flow for users
- a credential store for integration tokens and API tokens
- a provider graph built from inline definitions or packaged plugins
- optional runtimes and bindings
- an HTTP API, embedded web UI, and optional `/mcp` endpoint

## Runtime Primitives

| Primitive | What it does |
| --- | --- |
| `auth` | Authenticates the human caller of the platform. |
| `datastore` | Stores users, integration tokens, API tokens, staged connections, and plugin runtime state. |
| `secrets` | Resolves `secret://...` references and egress credential grants. |
| `integrations` | Become providers that expose operations. |
| `runtimes` | Long-lived background plugins that can call provider operations. |
| `bindings` | Extra HTTP routes such as inbound webhooks or outbound proxy surfaces. |
| `egress` | Governs what outbound HTTP calls bindings and upstreams can make. |
| `ui.plugin` | Replaces the embedded web app with a packaged UI bundle. |

## Startup Phases

### Load Config

Gestalt reads `config.yaml`, expands `${VAR}` placeholders, resolves relative
paths from the config directory, and validates the structure.

### Prepare Managed Artifacts

`gestaltd init` prepares artifacts referenced by `plugin.package`,
`plugin.source`, `runtimes.*.plugin`, and `ui.plugin`. It writes lock state next
to the config so `serve --locked` can start without mutating local files.

### Bootstrap Providers

Every integration becomes a provider. A provider can come from:

- inline declarative operations
- an inline OpenAPI, GraphQL, or upstream MCP surface
- a packaged plugin
- a hybrid package plus one or more spec-backed surfaces

### Start Extensions

After providers are ready, Gestalt starts runtimes and bindings. Bindings mount
extra routes under `/api/v1/bindings/...`.

### Serve HTTP And MCP

The server exposes:

- `/health`
- `/ready`
- `/api/v1/...`
- `/mcp` when at least one integration exposes tools
- the embedded UI or a packaged UI bundle

## Shared Provider Graph

Every surface reuses the same provider graph.

- The HTTP API calls provider operations directly.
- The UI discovers integrations and operations through the same API.
- `/mcp` turns provider catalogs into tool definitions.
- Runtimes invoke providers through the same broker used by HTTP requests.
- Bindings can forward into providers or proxy outbound HTTP with shared egress
  policy.

That shared broker is where Gestalt resolves connection mode, stored tokens,
refresh flows, instance selection, and audit metadata.

## Connections And Instances

Gestalt stores connection state by:

- subject
- integration
- connection name
- instance name

The server auto-selects a connection instance when there is exactly one match.
If multiple matches exist, callers must specify `_connection`, `_instance`, or
both.

## When To Reach For Each Primitive

- Use an inline integration when you only need a small API surface.
- Use a packaged provider when the logic belongs in code.
- Use a runtime when work starts from an external event or long-lived loop.
- Use a binding when you need an inbound webhook or a controlled proxy route.
- Use `/mcp` when the same provider graph should be available to MCP clients.

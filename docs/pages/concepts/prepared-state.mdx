# Prepared State

Gestalt can start in two modes:

- **Mutable startup**: `gestaltd serve`
- **Locked startup**: `gestaltd serve --locked`

Prepared state is what makes locked startup deterministic.

## What `gestaltd init` writes

Run:

```sh
gestaltd init --config ./config.yaml
```

`init` writes:

- `gestalt.lock.json` next to the config file
- prepared plugin and UI artifacts under `.gestaltd/` in the configured artifacts directory

For a typical deployment directory, that looks like:

```text
deploy/
  config.yaml
  gestalt.lock.json
  .gestaltd/plugins/...
```

If `server.artifacts_dir` or `--artifacts-dir` is set, the `.gestaltd/` directory is written there instead. The lockfile still lives next to the config file.

## What The Lockfile Is For

Pinned YAML config and lock state serve different purposes.

For a source-backed plugin, the YAML config says which plugin release to use:

```yaml
providers:
  example:
    from:
      source: github.com/acme/plugins/example
      version: 1.2.3
```

That identifies the intended plugin version. The lockfile records the prepared result of resolving that config:

- the exact release asset that was selected
- the digest of the downloaded archive
- the prepared manifest and executable or UI asset paths that `gestaltd` will use at runtime

In other words:

- the config declares intent
- the lockfile records the exact prepared artifact set

## Why Locked Startup Needs It

`gestaltd serve --locked` is intended for deterministic environments such as production images and Helm deployments. In locked mode, the server should not opportunistically resolve plugins or rewrite prepared state during startup.

The lockfile gives `gestaltd` a persisted record of what was prepared during `init`, so it can:

- verify that the prepared state still matches the config
- fail fast when prepared artifacts are missing or stale
- re-materialize the exact locked source artifact instead of re-resolving a release tag

Without lock state, `serve --locked` would have no durable way to distinguish:

- “this exact prepared artifact set matches the config”
- from “some files happen to exist under `.gestaltd/`”

## Typical Workflows

### Local development

Use mutable startup when convenience matters more than determinism:

```sh
gestaltd serve --config ./config.yaml
```

### Production and CI

Prepare artifacts first, then start from the prepared state only:

```sh
gestaltd init --config ./config.yaml
gestaltd serve --locked --config ./config.yaml
```

This is the recommended production pattern.

## Validation

To validate without starting the server:

```sh
gestaltd validate --init --config ./config.yaml
```

If prepared state already exists and you want validation to remain non-mutating:

```sh
gestaltd validate --config ./config.yaml
```

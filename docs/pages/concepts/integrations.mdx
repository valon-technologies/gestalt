# Integrations

Integrations are the unit of API exposure in Toolshed. Each integration declares where its definition comes from, how it authenticates, and which operations are allowed to execute.

## Three ways to define an integration

1. **OpenAPI spec** — set the `openapi` field to a URL or local path. Toolshed loads the spec at startup, extracts operations, and builds the provider automatically.
2. **Provider file** — write a standalone YAML file that declares the base URL, auth style, and operations manually. Place it in a directory listed in `provider_dirs`.
3. **Provider directory** — point `provider_dirs` at a directory containing multiple provider YAML files. Each file is registered as a separate provider at startup.

## What an operation looks like

- **Operation name** — the unique identifier used to invoke the operation. For OpenAPI-sourced operations, this is the `operationId` from the spec.
- **Description** — a human-readable summary. Comes from the OpenAPI `summary` field or the `allowed_operations` map value in config.
- **HTTP method and path** — the verb and URL path template for the upstream API call.
- **Parameters** — query parameters, path parameters, headers, and request body fields extracted from the spec or declared in the provider YAML.

## How OpenAPI loading works

When an integration includes an `openapi` field, Toolshed uses libopenapi to parse the spec and extract:

- The display name from the `info.title` field.
- The first server URL as the base URL for upstream requests.
- OAuth authorization and token URLs from the security scheme definitions.
- Operations by iterating every path and method, using the `operationId` as the operation name.

Operations without an `operationId` are skipped. If the spec includes security schemes, they inform the default auth configuration but can be overridden in the integration config.

## Manual auth versus OAuth

When `auth.type` is set to `manual`, the integration expects the caller to provide credentials directly. Toolshed stores them as-is and attaches them to upstream requests using the configured token parser hook. This is useful for APIs that use static API keys or other non-OAuth credentials.

For OAuth integrations, Toolshed manages the full authorization code flow. It generates the authorization URL, handles the callback, exchanges the code for tokens, encrypts them, and stores them in the datastore. On subsequent requests, the stored access token is attached to upstream calls.

## Hooks are where API-specific behavior lives

| Hook type | Purpose |
|---|---|
| `response_check` | Validate a successful-looking response body. |
| `token_parser` | Turn the stored token into auth headers. |
| `request_mutator` | Rewrite or augment outgoing requests before execution. |
| auth `response_hook` | Validate token exchange responses from OAuth providers. |

Hooks are registered by name in Go code and referenced by that name in config. They let you handle the quirks of individual APIs without changing the core Toolshed runtime.

## Current built-in hook registrations

- **slack_ok** — checks that the Slack API response body contains `"ok": true`. Used as both a `response_check` and an auth `response_hook`.
- **datadog_keys** — parses Datadog API and application keys from stored credentials and attaches them as request headers.
- **hex_response** — handles Hex-specific response validation for API calls.

## Restrictions and tradeoffs

Toolshed intentionally keeps the integration model simple. There is no plugin discovery protocol, no remote provider registry, and no dynamic loading at runtime. Integrations are declared in config and resolved at startup. This makes the system predictable and auditable at the cost of requiring a restart to add or change integrations.

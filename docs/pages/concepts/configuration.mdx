# Configuration

Gestalt is configured with a single YAML file. The file declares which platform components to use, which providers to enable, and how they connect to upstream services. `gestaltd` reads this file at startup and derives the entire running system from it.

## Config discovery

When `--config` is not specified, `gestaltd` searches for a config file in the following order:

1. The `GESTALT_CONFIG` environment variable
2. `./config.yaml`
3. `~/.gestaltd/config.yaml`
4. `/etc/gestalt/config.yaml`

If no config file is found, `gestaltd` generates a default local config at `~/.gestaltd/config.yaml` with anonymous auth and SQLite storage.

## Config loading

During loading, Gestalt expands `${ENV_VAR}` placeholders, applies defaults, validates the result, and resolves file paths relative to the config file's directory. After the platform's secret manager is initialized, `secret://` references are resolved.

## Top-level blocks

```yaml
server: {}
auth: {}
datastore: {}
secrets: {}
telemetry: {}
providers: {}
ui: {}
```

| Block | Role |
| --- | --- |
| `server` | Port, base URL, encryption key, and API token TTL. |
| `auth` | Platform authentication provider. |
| `datastore` | Storage backend for users, tokens, and runtime state. |
| `secrets` | Secret resolution backend for `secret://` references. |
| `telemetry` | Structured logging, traces, OTLP export, Prometheus scraping, and the built-in admin metrics UI. |
| `providers` | Integration definitions. |
| `ui` | Optional custom web UI plugin. |

## Defaults and required fields

Gestalt applies these defaults:

- `server.port`: `8080`
- `secrets.provider`: `env`
- `telemetry.provider`: `stdout`

These fields are required:

- `auth.provider`
- `datastore.provider`
- `server.encryption_key`

The encryption key is the root secret for the deployment. Gestalt derives all cryptographic material from it, including keys for token encryption and session signing.

## Prepared state

Gestalt supports two startup modes: mutable and deterministic.

In **mutable mode** (`gestaltd serve`), Gestalt may resolve remote dependencies, prepare managed artifacts, and regenerate lock state at startup. This is convenient for development.

In **deterministic mode**, you first run `gestaltd init` and then start the server with `gestaltd serve --locked`.

`gestaltd init` writes:

- `gestalt.lock.json` next to the config file
- prepared plugin and UI artifacts under `.gestaltd/` in the configured artifacts directory

For a typical deployment directory, that looks like:

```text
deploy/
  config.yaml
  gestalt.lock.json
  .gestaltd/providers/...
  .gestaltd/ui/...
```

If `server.artifacts_dir` or `--artifacts-dir` is set, the `.gestaltd/` directory is written there instead. The lockfile still lives next to the config file.

### What the lockfile is for

Pinned YAML config and lock state serve different purposes.

For a source-backed plugin, the YAML config says which plugin release to use:

```yaml
providers:
  example:
    from:
      source: github.com/acme/plugins/example
      version: 1.2.3
```

That identifies the intended plugin version. The lockfile records the prepared result of resolving that config:

- the exact release asset that was selected
- the digest of the downloaded archive
- the prepared manifest and executable or UI asset paths that `gestaltd` will use at runtime

The lockfile is organized by surface rather than by prefixed string keys:

```json
{
  "version": 2,
  "providers": {
    "example": {
      "fingerprint": "...",
      "source": "github.com/acme/plugins/example",
      "version": "1.2.3",
      "resolved_url": "https://github.com/...",
      "archive_sha256": "...",
      "manifest": ".gestaltd/providers/example/plugin.yaml",
      "executable": ".gestaltd/providers/example/artifacts/linux/amd64/provider"
    }
  },
  "ui": {
    "fingerprint": "...",
    "source": "github.com/acme/plugins/web",
    "version": "1.2.3",
    "resolved_url": "https://github.com/...",
    "archive_sha256": "...",
    "manifest": ".gestaltd/ui/plugin.yaml",
    "asset_root": ".gestaltd/ui/dist"
  }
}
```

In other words:

- the config declares intent
- the lockfile records the exact prepared artifact set

### Why locked startup needs it

`gestaltd serve --locked` is intended for deterministic environments such as production images and Helm deployments. In locked mode, the server should not opportunistically resolve plugins or rewrite prepared state during startup.

The lockfile gives `gestaltd` a persisted record of what was prepared during `init`, so it can:

- verify that the prepared state still matches the config
- fail fast when prepared artifacts are missing or stale
- re-materialize the exact locked source artifact instead of re-resolving a release tag

Without lock state, `serve --locked` would have no durable way to distinguish:

- “this exact prepared artifact set matches the config”
- from “some files happen to exist under `.gestaltd/`”

### Typical workflows

Use mutable startup when convenience matters more than determinism:

```sh
gestaltd serve --config ./config.yaml
```

For production and CI, prepare artifacts first and then start from the prepared state only:

```sh
gestaltd init --config ./config.yaml
gestaltd serve --locked --config ./config.yaml
```

To validate a config without starting the server:

```sh
gestaltd validate --init --config ./config.yaml
```

## Path resolution

All file paths in the config are resolved relative to the directory containing the config file, not the working directory of the process. This makes configs portable across local development, Docker mounts, and Kubernetes volumes.

## Minimal config

```yaml
server:
  port: 8080
  base_url: http://localhost:8080
  encryption_key: change-me

auth:
  provider: none

datastore:
  provider: sqlite
  config:
    path: ./gestalt.db

providers: {}
```

This is sufficient to start a working server. All other configuration is additive.

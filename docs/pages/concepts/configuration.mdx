# Configuration

Gestalt is configured with one YAML file. The file is small at the top level
and pushes most detail into `integrations.*.plugin`.

## Config Discovery

When you do not pass `--config`, Gestalt searches in this order:

1. the `GESTALT_CONFIG` environment variable
2. `./config.yaml`
3. `~/.gestalt/config.yaml`
4. `/etc/gestalt/config.yaml`

If you run bare `gestaltd` and no config exists, Gestalt writes a new local file
to `~/.gestalt/config.yaml`.

## YAML Rules

- `${VAR}` expansion happens before YAML decoding.
- `${VAR_FILE}` is also supported. The file contents replace the placeholder
  with trailing newlines trimmed.
- Unknown fields are rejected.
- Relative paths are resolved from the directory that contains the config file.
- `server.port` defaults to `8080`.
- `secrets.provider` defaults to `env`.
- `telemetry.provider` defaults to `stdout`.

## Top-Level Shape

```yaml
server: {}
auth: {}
datastore: {}
secrets: {}
telemetry: {}
integrations: {}
runtimes: {}
bindings: {}
egress: {}
ui: {}
```

This is the top-level map shape, not a complete file.

The full field reference lives at [Config File](/reference/config-file).

## Local And Locked Startup

Gestalt supports two operating modes.

### Mutable Local Startup

```sh
gestaltd --config ./config.yaml
```

This path is best for local development. If packaged plugins are stale or
missing, Gestalt can prepare them before serving.

### Locked Startup

```sh
gestaltd init --config ./config.yaml
gestaltd serve --locked --config ./config.yaml
```

This path is best for CI and production. `serve --locked` refuses to continue
when prepared artifacts are missing or stale.

## What `init` Actually Writes

`gestaltd init` prepares managed artifacts for:

- `integrations.*.plugin.package`
- `integrations.*.plugin.source`
- `runtimes.*.plugin.package`
- `runtimes.*.plugin.source`
- `ui.plugin.package`
- `ui.plugin.source`

It writes:

- `gestalt.lock.json`
- `.gestalt/plugins/...` for prepared packages

Inline integrations do not need packaged artifacts, so `init` is usually a
cheap no-op apart from writing the lockfile.

## Relative Paths

These values are resolved from the config directory:

- `icon_file`
- `plugin.command` when it is a relative executable path
- `plugin.package`
- `plugin.openapi` when it points at a local file
- `plugin.graphql_url` and `plugin.mcp_url` when they are local file paths

## Secrets And Environment

There are two separate mechanisms:

- `${VAR}` and `${VAR_FILE}` for config-time interpolation
- `secrets.provider` plus `secret://name` references for runtime secret lookup

Use config interpolation for deployment-specific values like `DATABASE_URL`.
Use `secret://...` when a feature expects a secret reference at runtime, such as
egress credential grants.

## Base URL

`server.base_url` is optional but important whenever the server needs to emit
absolute callback URLs or set secure cookies correctly. When it is set:

- platform login callbacks use it
- integration OAuth callbacks use it
- cookies are marked secure for `https://...` base URLs

## The Smallest Useful Config

With one local OpenAPI file:

```yaml
openapi: 3.0.3
info:
  title: HTTPBin
  version: 1.0.0
servers:
  - url: https://httpbin.org
paths:
  /ip:
    get:
      operationId: get_ip
      responses:
        "200":
          description: OK
```

```yaml
server:
  port: 8080
  encryption_key: change-me

auth:
  provider: none

datastore:
  provider: sqlite
  config:
    path: ./gestalt.db

integrations:
  example:
    plugin:
      openapi: ./openapi/httpbin.yaml
      openapi_connection: public
      default_connection: public
      connections:
        public:
          mode: none
          auth:
            type: none
```

That is enough to boot a local server, expose an operation over HTTP, and make
the same operation available as an MCP tool.

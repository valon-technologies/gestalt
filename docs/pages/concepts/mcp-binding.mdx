# MCP Binding

Toolshed exposes integrations through two transport bindings: the HTTP API server (`toolshed serve`) and the MCP stdio server (`toolshed mcp`). Both bindings share the same config, providers, datastore, and auth layer. The difference is how clients connect.

## What MCP is

The [Model Context Protocol](https://modelcontextprotocol.io) (MCP) is an open standard for connecting AI assistants to external tools. An MCP server advertises a list of tools and handles tool calls from a connected client. Toolshed's MCP binding maps each allowed integration operation to an MCP tool.

## How tool names are derived

Each MCP tool name is built from the provider name and operation ID, joined by an underscore:

```
{provider}_{operation}
```

For example, the `search_issues` operation on the `linear` provider becomes the tool `linear_search_issues`.

If you set `tool_name_prefix` in the `mcp` config block, that string is prepended to every tool name. A prefix of `ts_` would produce `ts_linear_search_issues`.

## How auth works in stdio mode

The HTTP API server authenticates callers via session tokens or API tokens in the `Authorization` header. The stdio MCP server cannot use HTTP headers, so it reads a `TOOLSHED_API_KEY` environment variable at startup. The API key is validated against the datastore and resolved to a user identity before any tool calls are processed. All tool invocations run as that user.

## Relationship to the HTTP API

The MCP binding is not a replacement for the HTTP API. It is an alternative transport for the same operations:

- Same providers and integration definitions from config.
- Same datastore for users, tokens, and credentials.
- Same invocation path: the broker resolves the caller's stored credential for the target integration, refreshes it if expired, and executes the upstream API call.
- Same allow-list enforcement: only operations listed in `allowed_operations` are callable.

The MCP binding adds an `mcp.providers` filter so you can expose a subset of integrations to MCP clients without affecting the HTTP API.

## Tool metadata

Toolshed populates MCP tool annotations automatically. GET and HEAD operations are marked as read-only. PUT operations are marked as idempotent. DELETE operations are marked as destructive. Input schemas from OpenAPI specs are passed through as the tool's JSON Schema, giving MCP clients type information for parameters.

## What is not in the MVP

- **Streamable HTTP transport** -- the binding uses stdio only. Streamable HTTP MCP support is planned but not yet implemented.
- **Hot-reload** -- adding or removing integrations requires restarting the MCP server.
- **Auto-spawning** -- Toolshed does not manage the MCP server process lifecycle. Your MCP client (Claude Code, Claude Desktop, etc.) starts and stops it.

# Plugin Release Archives

Gestalt uses one release archive format for:

- provider plugins
- web UI bundles

Each release archive is described by a manifest file named one of:

- `plugin.yaml`
- `plugin.yml`
- `plugin.json`

## How Plugins Enter A Config

There are two ways to point at a plugin from `config.yaml`.

### `source.path`

Use a local source tree directly.

```yaml
providers:
  example:
    from:
      source:
        path: ./plugin.yaml
    config:
      greeting: hello
```

This is best for local development.

### `source.ref`

Resolve a versioned plugin source during `init`.

In config, that looks like `source.ref: github.com/acme/plugins/example` plus
`source.version: 1.2.3`.

This is the cleanest path for reusable, versioned plugins.

## What `init` Does For Published Plugins

`gestaltd init` resolves and prepares:

- `from.source`
- `ui.plugin.source`

It writes lock state (`gestalt.lock.json`) and extracts prepared artifacts under `.gestaltd/`. See [Configuration](/concepts/configuration) for the deployment model around `init` and `serve --locked`.

## Release Command

Build a cross-platform release:

```sh
gestaltd plugin release --version 0.1.0
```

Run `plugin release` from the plugin source directory. It compiles the plugin,
produces release archives, and writes a checksums file.

## Declarative-Only Archives

A provider manifest can be declarative-only. In that case the release archive contains a
manifest and optional schemas or assets, but no executable artifact is required.

That is useful when you want to ship:

- a reusable declarative REST provider
- a reusable OpenAPI-backed integration
- a reusable web UI bundle

## Executable Archives

Executable provider release archives add `artifacts` plus
`entrypoints.provider.artifact_path` to the manifest. `gestaltd plugin release`
builds that structure for you.

## Release Archive Layout

Typical contents:

- `plugin.yaml`
- `artifacts/...` for compiled binaries
- `schemas/config.schema.json` or `schemas/config.schema.yaml` for provider config validation
- `assets/...` for icons or web assets

The exact layout is controlled by the manifest.

# Observability

Gestalt ships with structured logging, health endpoints, readiness checks, a
Prometheus scrape endpoint, and optional OpenTelemetry export.

## Health Endpoints

| Path | Meaning |
| --- | --- |
| `/health` | Process liveness. Returns `200` when the server is up. |
| `/ready` | Startup readiness. Returns `503` until providers and the datastore are ready. |

## Default Telemetry

If you omit the `telemetry` block, Gestalt uses:

```yaml
telemetry:
  provider: stdout
```

That gives you structured logs without external collectors.

By default, `stdout` also exposes a Prometheus-compatible `/metrics` endpoint.
Traces remain disabled unless you switch to `telemetry.provider: otlp`.

## OTLP Export

Use `telemetry.provider: otlp` to export traces, metrics, and logs.

```yaml
telemetry:
  provider: otlp
  config:
    endpoint: otel-collector:4317
    protocol: grpc
    service_name: gestaltd
    traces:
      sampling_ratio: 1.0
    metrics:
      interval: 60s
    logs:
      level: info
```

Supported OTLP config includes:

- `endpoint`
- `protocol`
- `service_name`
- `insecure`
- `headers`
- `resource_attributes`
- trace sampling
- metric export interval
- Prometheus enablement
- log level

If you want traces and metrics in OTLP but want application logs to remain on
stdout for the host platform to collect, set:

```yaml
telemetry:
  provider: otlp
  config:
    endpoint: otel-collector:4317
    protocol: grpc
    logs:
      exporter: stdout
      format: json
      level: info
```

## Emitted Metrics

The built-in metric surface currently covers:

- exporter metadata from the Prometheus bridge
- inbound HTTP requests handled by `gestaltd`
- broker operation execution
- integration auth, connect, and token refresh flows
- outbound gRPC calls from `gestaltd` to plugin processes

Gestalt documents metric names using their OpenTelemetry instrument names. The
Prometheus scrape endpoint translates those names to Prometheus-style metric
families:

- dots become underscores
- counters gain a `_total` suffix
- units become suffixes like `_seconds`, `_milliseconds`, or `_bytes`
- histograms appear as `_bucket`, `_sum`, and `_count` series under the same
  Prometheus family

For example, `gestaltd.operation.count` is exposed as
`gestaltd_operation_count_total`, and `gestaltd.operation.duration` is exposed
as the `gestaltd_operation_duration_seconds` histogram family.

### Prometheus Exporter Metadata

| OTel name | Prometheus family | Type | Meaning |
| --- | --- | --- | --- |
| — | `target_info` | Gauge | Exporter-generated metadata metric built from the server resource attributes, including `service_name`. |

### Broker Operation Metrics

These are Gestalt-specific metrics recorded around `broker.invoke`.

| OTel instrument | Prometheus family | Type | Meaning |
| --- | --- | --- | --- |
| `gestaltd.operation.count` | `gestaltd_operation_count_total` | Counter | Counts operation invocations. |
| `gestaltd.operation.error_count` | `gestaltd_operation_error_count_total` | Counter | Counts operation invocations that fail. |
| `gestaltd.operation.duration` | `gestaltd_operation_duration_seconds` | Histogram | Measures operation invocation duration. |

These metrics use low-cardinality attributes:

- `gestalt.provider`: the integration key
- `gestalt.operation`: the catalog operation id
- `gestalt.transport`: how the operation is implemented: `plugin`, `rest`, or `mcp-passthrough`
- `gestalt.connection_mode`: the provider auth mode: `none`, `user`, `identity`, or `either`

When a request references an unknown provider or operation, `gestaltd` records
`unknown` for the missing metric attributes instead of using raw user input.

### Connection Lifecycle Metrics

These are Gestalt-specific metrics recorded around integration auth and
credential refresh flows.

| OTel instrument | Prometheus family | Type | Meaning |
| --- | --- | --- | --- |
| `gestaltd.oauth.callback.count` | `gestaltd_oauth_callback_count_total` | Counter | Counts integration OAuth callback outcomes. |
| `gestaltd.oauth.token_refresh.count` | `gestaltd_oauth_token_refresh_count_total` | Counter | Counts OAuth token refresh attempts actually performed by `gestaltd`. |

These metrics use low-cardinality attributes:

- `gestalt.provider`: the integration key
- `gestalt.result`: `success` or `error`
- `gestalt.connection_mode`: the provider auth mode on `gestaltd.oauth.token_refresh.count`

### Current Non-Goals

The built-in `gestaltd` metric surface still does not emit:

- per-operation upstream HTTP `status_code` or `status_class` metrics
- deduplicated DAU, WAU, or MAU analytics for users or integrations

Those need separate semantic decisions or an analytics pipeline instead of more
process-local counters.

### HTTP Server Metrics

These come from the OpenTelemetry HTTP middleware wrapped around the server's
main router. Because the whole router is instrumented once, these metrics cover
API traffic, health and readiness checks, the embedded admin UI, and scrapes of
`/metrics` itself.

| OTel instrument | Prometheus family | Type | Meaning |
| --- | --- | --- | --- |
| `http.server.request.body.size` | `http_server_request_body_size_bytes` | Histogram | Measures inbound request body size. |
| `http.server.response.body.size` | `http_server_response_body_size_bytes` | Histogram | Measures response body size. |
| `http.server.request.duration` | `http_server_request_duration_seconds` | Histogram | Measures end-to-end request duration. |

These metrics use standard OpenTelemetry HTTP server semantic attributes such as
request method, response status, scheme, host, protocol, and route information
when available.

If the process is started with
`OTEL_SEMCONV_STABILITY_OPT_IN=http/dup`, the underlying HTTP instrumentation
also emits the older legacy HTTP metric families alongside the current ones.

### Plugin gRPC Client Metrics

These come from the OpenTelemetry gRPC client stats handler used for outbound
gRPC traffic from `gestaltd` to plugin processes. They only apply to
plugin-backed providers.

| OTel instrument | Prometheus family | Type | Meaning |
| --- | --- | --- | --- |
| `rpc.client.duration` | `rpc_client_duration_milliseconds` | Histogram | Measures outbound plugin RPC duration. |
| `rpc.client.request.size` | `rpc_client_request_size_bytes` | Histogram | Measures outbound plugin request message size. |
| `rpc.client.response.size` | `rpc_client_response_size_bytes` | Histogram | Measures inbound plugin response message size. |
| `rpc.client.requests_per_rpc` | `rpc_client_requests_per_rpc` | Histogram | Measures request message count per RPC. |
| `rpc.client.responses_per_rpc` | `rpc_client_responses_per_rpc` | Histogram | Measures response message count per RPC. |

These metrics use standard OpenTelemetry gRPC client semantic attributes such as
`rpc.system=grpc`, `rpc.service`, and `rpc.method`.

## Prometheus Scrape Endpoint

When telemetry metrics are enabled, Gestalt exposes a Prometheus scrape endpoint
at `/metrics`.

- `telemetry.provider: stdout` keeps metrics local and serves them directly.
- `telemetry.provider: otlp` exports metrics over OTLP and also serves `/metrics`
  locally unless you disable the Prometheus bridge.
- `telemetry.provider: noop` disables Prometheus scraping; `/metrics` returns a
  clear unavailable response so the admin UI can surface that state.

`/metrics` is an authenticated Prometheus text endpoint. The embedded admin UI
visualizes that same scrape surface instead of using a separate metrics API.

The Prometheus endpoint and OTLP export hang off the same meter provider and
include the HTTP middleware metrics, broker metrics, plugin gRPC client
metrics, and exporter metadata described above.

The admin UI intentionally stays basic:

- summary cards and lightweight charts derived from `/metrics`
- raw Prometheus text available in-page for inspection
- periodic browser refreshes
- no persisted history
- no cross-replica aggregation

It is designed for built-in operator visibility, not as a replacement for a
full observability backend.

## What Gets Logged

Gestalt logs:

- config loading and startup
- provider readiness
- auth and connection flow failures
- invocation failures
- datastore warnings

When OTLP is enabled, the server routes structured logs through the OpenTelemetry
log bridge.

# HTTP API

The server surface is intentionally small. Most of the power comes from the integrations you load into it.

## Unauthenticated endpoints

| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | Health check. |
| POST | `/api/v1/auth/login` | Start platform login and receive a browser URL. |
| GET | `/api/v1/auth/login/callback` | Complete platform login and optionally issue a session token. |

## Authenticated endpoints

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/v1/integrations` | List integrations currently loaded into the server. |
| GET | `/api/v1/integrations/{name}/operations` | List the operations exposed by an integration. |
| GET or POST | `/api/v1/{integration}/{operation}` | Execute an integration operation. |
| POST | `/api/v1/auth/start-oauth` | Start an integration OAuth flow. |
| GET | `/api/v1/auth/callback` | Complete integration OAuth and store the resulting token. |
| POST | `/api/v1/tokens` | Create an API token for the current user. |
| GET | `/api/v1/tokens` | List API tokens for the current user. |
| DELETE | `/api/v1/tokens/{id}` | Revoke an API token. |

## Execution semantics

- **GET** requests pass parameters as query string key-value pairs.
- **POST** requests pass parameters as a JSON body.
- Path placeholders like `{integration}` and `{operation}` are resolved from the URL. Any remaining parameters declared by the operation are sent as query or body values depending on the method.

## Authentication header

```
Authorization: Bearer <session-token-or-api-token>
```

## Example request

```sh
curl -X POST http://localhost:8080/api/v1/slack/chat_postMessage \
  -H "Authorization: Bearer $TOOLSHED_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"channel":"C123","text":"hello from toolshed"}'
```
